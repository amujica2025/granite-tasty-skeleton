\# Journal Agent — Inputs



\## Product Inputs

The Journal Agent receives:

\- user requirement that Journal be user-focused, not dev-focused

\- user requirement that Journal support both live and retrospective context

\- user requirement that Journal take in as much meaningful context as possible while leaving the smallest footprint possible

\- user interest in capturing ideas, targets, worksheets, and reviews

\- possible future desire to capture broader activity context (searches, charts, surface scans, etc.)



\## Data Inputs

Journal may consume:

\- symbol

\- underlying

\- strategy name

\- target

\- thesis

\- risk notes

\- scanner state summary

\- selected scanner candidates

\- artifact session summary

\- watchlist symbol context

\- alert events

\- trade entry / exit metadata

\- worksheet values

\- timestamps

\- tags

\- user notes



\## Event Inputs

Journal should be able to accept event-based capture from:

\- trade idea creation

\- pre-trade worksheet completion

\- alert trigger

\- scanner shortlist selection

\- artifact save/share

\- pre-exit review

\- post-trade recap



\## Storage Inputs

Journal should support:

\- lightweight local persistence

\- optional structured event storage

\- optional richer snapshot storage on meaningful triggers only



\## Structural Inputs

The Journal Agent should assume:

\- Journal already exists as a popup/panel concept

\- Journal should not become a raw developer event dump

\- Journal should default to low-footprint structured logging
